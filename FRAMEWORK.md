## 1. 框架概述

**Research Innovation Scout** 是一个学科无关的通用框架，帮助研究者从一个模糊的研究方向、技术名词或关键词出发，经过系统化的五步流程，获得**有文献支撑的、可选的科研创新方向建议**。

### 核心价值

| 维度 | 传统做法 | 本框架做法 |
|------|---------|-----------|
| 起点 | 依赖个人经验和阅读积累 | 任意模糊输入即可启动 |
| 证据 | 常凭直觉，缺乏系统追溯 | 每一步均要求文献/场景证据 |
| 覆盖面 | 被个人阅读范围限制 | 结构化扩展，覆盖深化/迁移/无人区/嫁接四类方向 |
| 可靠性 | 可能产生凭空编造的"创新" | 防幻觉设计，无证据则明确告知 |
| 学科限制 | 通常绑定特定学科工具 | 学科无关，自动识别领域 |

### 框架定位

> **本框架只做一件事：从输入到可选创新方向的决策支持报告。**
>
> 不做的事情：代码实现、实验设计、论文撰写、投稿建议。

## 2. 适用场景

### 2.1 典型用户画像

- **博士生/博士后**：进入新领域，需要快速找到有发表价值的研究问题
- **跨学科研究者**：将一个领域的方法迁移到另一个领域，需要定位切入点
- **导师/项目负责人**：需要为课题组探索新方向，希望有系统化的文献支撑
- **企业研究员**：需要评估某个技术方向的前沿进展和创新空间

### 2.2 输入类型示例

| 类型 | 示例 |
|------|------|
| 宽泛领域 | "人工智能在材料科学中的应用" |
| 技术方法 | "Graph Neural Networks" |
| 具体问题 | "如何提高小样本学习的泛化能力" |
| 跨领域想法 | "把强化学习用到药物分子设计里" |
| 单一论文/标题 | 提供一篇论文的 DOI 或标题，以此为种子展开 |

### 2.3 不适用的情况

- 用户已经明确知道要做什么实验、写什么论文（此时需要的是实验设计/写作工具）
- 需要的是文献综述写作（本框架可输出素材，但不组织成综述格式）
- 纯工程/产品开发问题（不涉及研究贡献）

## 3. 输入与输出说明

### 3.1 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `user_input` | string | 是 | 研究兴趣描述，支持模糊/宽泛/具体 |
| `domain_hint` | string | 否 | 用户指定的学科领域（如未提供则自动识别） |
| `scope` | enum | 否 | `narrow` / `broad`，默认 `broad`；影响文献检索广度 |
| `time_range` | string | 否 | 如 `2019-2024`，默认近 5 年 |
| `language` | string | 否 | 检索语言偏好，默认不限制 |

### 3.2 输出规范

最终输出为一封结构化的**决策支持报告**（详见[步骤 5](#10-步骤-5输出决策支持报告)），包含：

- **执行摘要**：输入理解 + 场景确认 + 文献概况 + 方向概览
- **场景证据卡**：找到的真实科研场景及其支撑证据
- **文献证据链**：筛选后的关键文献清单及四维拆解摘要
- **创新方向建议**：按四类方向（深化优化、技术迁移、无人区、问题嫁接）组织，每个方向附：
  - 核心想法
  - 文献支撑（具体到哪篇文献的哪个未解决问题/局限）
  - 新颖度评估（高/中/低 + 理由）
  - 可行性评估（高/中/低 + 前提条件）
  - 潜在风险评估
- **方法论图谱**：当前场景下主要方法的分类与关系
- **开放问题池**：来自多篇文献的未解决问题汇总

## 5. 步骤 0：输入分类与问题化引导

### 5.1 目标

将用户任意形式的输入归类为三种基本类型之一，并根据类型进行差异化引导，最终输出一个**可操作的研究问题场景**，为步骤 1 提供清晰的起点。

### 5.2 三种输入类型

#### 类型 A：具体研究问题
- **特征**：用户输入已包含明确的研究 gap 或 hypothesis
- **示例**："如何在低资源语言上实现 zero-shot 跨语言迁移？"
- **处理**：直接提取核心问题作为场景起点，标记`input_type = "specific_question"`

#### 类型 B：技术/方法导向
- **特征**：用户指定了某一技术、算法或方法，但未明确要解决什么场景下的什么问题
- **示例**："Graph Neural Networks"、"Transformer"、"CRISPR"
- **处理**：需要**反向追问场景**——"这个方法主要被用于解决哪类问题？在什么场景下最活跃？"，标记`input_type = "method_oriented"`

#### 类型 C：宽泛领域/方向
- **特征**：输入是宏观领域、跨学科概念或模糊方向
- **示例**："AI for Science"、"碳中和"、"脑机接口"
- **处理**：需要**子领域聚焦**——将该领域拆解为若干活跃子方向，并请用户选择或确认聚焦范围，标记`input_type = "broad_domain"`

### 5.3 操作流程

```
1. 接收用户输入
2. 分析输入的明确程度
3. 判断输入类型（A/B/C）
4. 分流处理：
   ├── 类型 A → 直接提取 RQ，进入步骤 1
   ├── 类型 B → 反向追问场景，收窄后进入步骤 1
   └── 类型 C → 拆解子方向，用户确认后进入步骤 1
5. 输出：一个明确的 research question 场景描述 + input_type 标记
```

### 5.4 分支情况与异常处理

| 情况 | 处理方式 |
|------|---------|
| 输入过于模糊，无法判断类型 | 请求用户补充信息，提供 2-3 个可能的解读方向供选择 |
| 用户输入跨多个领域 | 分别标注，优先选择用户最明确的那一个，其余作为备选记录 |
| 用户直接提供论文 DOI | 归为类型 B 的特殊情况：以该论文的方法/问题为种子，反向扩展场景 |

### 5.5 输出示例

```markdown
## 步骤 0 输出
- **输入类型**: method_oriented (类型 B)
- **原始输入**: "Graph Neural Networks"
- **场景聚焦**: GNN 在分子属性预测（molecular property prediction）场景中的应用
- **核心研究问题**: 如何利用图神经网络提高小样本下的分子属性预测精度？
- **逻辑**: GNN 在分子建模中应用广泛，小样本场景是当前瓶颈之一
```

## 6. 步骤 1：场景生成与真实性校验

### 6.1 目标

基于步骤 0 输出的研究问题场景，找到**真实、活跃的科研场景**，并**提供可验证的证据**。核心原则：**找不到就明确说明，绝不编造**。

### 6.2 操作流程

```
1. 根据步骤 0 的场景描述，生成候选科研场景列表（3-5 个）
2. 对每个候选场景，寻找真实性证据：
   ├── 是否有近 3 年内的综述文章（survey/review）专门涵盖该场景？
   ├── 是否有相关的 workshop / shared task / challenge？
   ├── 是否有公认的 benchmark 或数据集？
   ├── 顶级会议/期刊是否有相关 session 或 track？
   └── 是否有活跃的开源项目或社区？
3. 对每个候选场景进行"证据质量"评级
4. 筛选出证据质量最高的 1-3 个场景
5. 如果所有候选场景都无证据支撑 → 输出"未找到真实活跃场景"
```

### 6.3 证据质量评级

| 等级 | 标准 | 示例 |
|------|------|------|
| 🟢 强证据 | 近 2 年内有专门综述 + benchmark/workshop + 多篇顶会论文 | "分子性质预测 → 有 MIT 的 MoleculeNet benchmark、多篇 NeurIPS/ICLR 论文" |
| 🟡 中等证据 | 有综述但较旧(>2 年)，或有追踪文章但不成体系 | "有 2020 年的 survey，近年有零散后续工作" |
| 🔴 弱证据 | 仅有零星论文，无综述或 benchmark | "仅能找到 2-3 篇相关文章，未见综述" |
| ⚪ 无证据 | 未找到任何可信来源支撑 | 直接告知用户，终止该场景的后续分析 |

### 6.4 场景证据卡模板

每个通过校验的场景，按以下模板输出证据卡：

```markdown
### 场景: [场景名称]

**证据等级**: 🟢 强证据

**综述支撑**:
- [综述标题] ([年份], [来源])
- 链接/DOI: [如有]

**Benchmark/数据集**:
- [名称]: [简要描述]

**学术活动**:
- Workshop: [名称] @ [会议] ([年份])
- Challenge: [名称] ([年份])

**活跃度指标**:
- 顶会相关论文数 (近 2 年): [数量或描述]
- 主要发表场所: [会议/期刊名称]
```

### 6.5 分支情况与异常处理

| 情况 | 处理方式 |
|------|---------|
| 找到 3+ 个强证据场景 | 按活跃度排序，优先分析前 2-3 个 |
| 仅有中等/弱证据场景 | 选择最好的 1-2 个继续，但在报告中标注"证据较弱" |
| 所有候选场景均无证据 | **明确告知用户**："在当前检索范围内未找到有充分证据支撑的活跃科研场景。建议：(1) 扩大检索范围/时间窗口；(2) 调整研究问题表述；(3) 考虑相邻领域。" |
| 找到的场景与用户预期不符 | 列出实际找到的场景，让用户确认是否接受或重新调整输入 |

### 6.6 关键约束

- **禁止虚构场景**：所有场景必须有至少一条可追溯到具体来源的证据（综述、benchmark、workshop 等）
- **禁止编造论文**：综述/论文标题和作者必须与真实文献一致
- **证据可验证**：尽量提供 DOI、会议名称、年份等可被用户自行验证的信息

## 7. 步骤 2：文献检索与筛选

### 7.1 目标

在步骤 1 确认的场景下，系统化地检索并筛选出**高质量、高相关度的核心文献集合**，形成后续分析的文献池。

### 7.2 检索策略

#### 7.2.1 种子文献定位（第一步）

```
1. 基于场景关键词，检索 2-3 篇种子文献
   种子文献标准：
   - 优先选择该场景的权威综述 (survey/review)
   - 其次选择高被引或顶会/顶刊的开创性工作
   - 必须能直接证明该场景的真实性
2. 记录种子文献的元信息（标题、作者、年份、来源、被引数）
```

#### 7.2.2 前后向扩展（第二步）

```
以种子文献为起点：

前向扩展 (Forward Search)：
├── 谁引用了这篇种子文献？
├── 近 2-3 年内引用该文献的工作（限制时间窗口）
└── 筛选出与该场景高度相关的后续工作

后向扩展 (Backward Search)：
├── 种子文献引用了哪些工作？
├── 追溯该场景的奠基性工作
└── 筛选出与当前问题设定相关的经典文献
```

### 7.3 筛选硬标准

| 维度 | 标准 | 说明 |
|------|------|------|
| **时间** | 默认近 5 年（可配置）；综述可放宽至近 3 年 | 经典奠基性工作可适当放宽，但需标注 |
| **来源** | 优先：CCF 推荐列表/A类会议期刊、中科院分区 Q1-Q2 | 非学术来源（博客、技术报告）仅在无正式论文时作为辅助参考，且需标注 |
| **相关度** | 标题/摘要必须与场景直接相关 | 仅引用了该场景但并非主要工作的文献应排除 |
| **类型** | 优先：survey/review、experimental paper、benchmark paper | 排除：opinion piece、未发表的 preprint（除非无替代且明确标注） |

### 7.4 文献池规模控制

| 场景证据等级 | 目标文献数 | 说明 |
|-------------|-----------|------|
| 🟢 强证据 | 15-25 篇 | 场景活跃，可在上限内精选 |
| 🟡 中等证据 | 8-15 篇 | 文献有限，降低数量标准但保持质量标准 |
| 🔴 弱证据 | 5-10 篇 | 如实反映文献稀缺情况 |

### 7.5 文献质量加权

对每篇进入文献池的论文进行质量标注：

```markdown
- **Q1 (高)**: 顶会/顶刊 + 高被引 (>50) 或近 2 年发表
- **Q2 (中)**: 正规学术期刊/会议，有一定引用或方法上有特色
- **Q3 (可参考)**: 相关性高但来源较弱（如 workshop paper、arXiv preprint），需在报告中显式标注
```

### 7.6 分支情况与异常处理

| 情况 | 处理方式 |
|------|---------|
| 检索结果 > 30 篇 | 按质量加权排序，取 top-K（按目标文献数）；记录被排除的数量和理由 |
| 检索结果很少（< 5 篇） | **警告用户**："在该场景下找到的文献较少（仅 X 篇），创新点分析可能不够全面。这可能意味着：(1) 该方向确实很新，文献积累不足；(2) 检索词或范围需要调整。是否继续？" |
| 种子文献无法确定 | 使用更宽泛的关键词搜索综述，如仍找不到 → 该场景证据降级 |
| 前后向扩展后无有效新增 | 以种子文献集合为最终文献池，标注"扩展未产生额外有效文献" |

### 7.7 检索工具使用说明

> **在封装为 AI Skills 时，此步骤可接入实际的文献检索 API。** 当前阶段，此处仅描述方法论和筛选逻辑。可接驳的工具包括但不限于：
> - Semantic Scholar API
> - arXiv API
> - DBLP
> - PubMed（生物医学）
> - IEEE Xplore
> - Google Scholar（需注意反爬限制）

## 8. 步骤 3：单篇文献四维拆解

### 8.1 目标

对步骤 2 文献池中的每一篇论文，从四个维度进行结构化深度分析，提取可用于创新的关键信息。**核心要求：严格区分作者自述（explicit）与读者推断（inferred）。**

### 8.2 四个维度

#### 维度 1：解决的问题（Problem）

```
分析要点：
- 该论文试图解决什么研究问题？
- 问题的具体设定是什么？（任务定义、约束条件、评价指标）
- 该问题为什么重要？（作者如何论证问题的重要性）
- 与该问题密切相关的先前工作有哪些？

输出格式：
- 问题定义: [一句话概括]
- 任务设定: [具体描述]
- 重要性论证: [作者的核心论点]
- 关键相关工作的差距: [作者指出的 prior work 的不足]
```

#### 维度 2：方法范式（Methodology）

```
分析要点：
- 该方法属于什么范式？（如：监督学习/自监督/强化学习、实证/理论/仿真等）
- 方法的核心机制是什么？（不是复述所有细节，而是提取核心 idea）
- 与同类方法的关键差异是什么？
- 方法有哪些假设或前提条件？

输出格式：
- 方法范式: [分类标签]
- 核心机制: [1-2 句话描述核心思路]
- 关键差异: [与最相近方法的不同之处]
- 假设/前提: [列举方法成立的前提条件]
```

#### 维度 3：核心贡献（Contribution）

```
分析要点：
- 作者声称的贡献有哪些？（技术贡献、经验发现、理论证明、数据集/benchmark 等）
- 贡献的实质是什么？（区分"我们首次做了X"和"我们改进了X的Y方面"）
- 实验/评估是否充分支撑了贡献声明？

输出格式：
- 作者声称的贡献: [列表]
- 贡献类型: [技术方法 / 经验发现 / 理论分析 / 资源贡献]
- 实验支撑力度: [充分 / 较充分 / 存在局限]
```

#### 维度 4：未解决的问题（Unresolved Issues）⚠️ 关键维度

```
分析要点（严格分类）：

【作者自述的局限性 / 未来工作】(Explicit)
- 作者在论文中明确提出的 limitation
- 作者在 conclusion/future work 中提出的方向
- 引用来源: [段落/行号]

【读者可推断的开放问题】(Inferred — 需明确标注)
- 基于方法假设的薄弱环节推断的问题
- 基于实验设计缺陷推断的 gap
- 基于该场景发展脉络推断的空白
- 推理逻辑: [从什么观察到什么推断]

⚠️ 严禁将推断问题伪造成作者自述的问题！

输出格式：
- 作者明确提出的未解决问题:
  1. [问题描述] (来源: §X, Conclusion)
  2. ...

- 可推断的开放问题 (⚠️ 推断):
  1. [问题描述] (推理逻辑: ...)
  2. ...
```

### 8.3 操作流程

```
对于文献池中的每篇论文：
1. 阅读标题、摘要、引言 → 初步判断四个维度
2. 精读方法部分 → 完善维度 2
3. 阅读实验/结论部分 → 完善维度 1、3、4
4. 形成四维拆解卡片
5. 标注文献质量等级 (Q1/Q2/Q3)
```

### 8.4 分支情况与异常处理

| 情况 | 处理方式 |
|------|---------|
| 论文为综述/立场文章，无具体方法 | 维度 2 改为"覆盖的方法范式"，维度 3 改为"综述的核心发现/观点"，维度 4 改为"综述指出的开放问题" |
| 论文信息不完整（如只有摘要可获取） | 标注"信息受限"，仅拆解可获取的部分，注明缺失维度 |
| 作者未明确提及局限性 | 维度 4 的"作者自述"部分标注"无"，仅填写推断部分并明确标注 |
| 推断部分超出合理范围（过度解读） | 删除超出部分，仅保留有逻辑链支撑的推断 |

## 9. 步骤 4：跨文献综合分析

### 9.1 目标

在完成文献池中所有文献的四维拆解后，进行跨文献的综合分析：

1. **合并开放问题池**
2. **绘制方法论图谱**
3. **生成四类创新方向建议**

### 9.2 合并开放问题池

```
操作：
1. 收集所有文献的"维度 4：未解决的问题"
2. 将相似问题归类（使用主题聚类，而非简单去重）
3. 区分"作者自述的开放问题"与"推断的开放问题"
4. 标注每个问题的文献来源（可追溯到具体论文）
5. 标注每个问题被提出的频次（多篇论文共同指出 → 高共识度）

输出格式：
| 编号 | 开放问题 | 类型 | 来源文献 | 共识度 | 备注 |
|------|---------|------|---------|--------|------|
| OP-01 | ... | 作者自述 | [Paper A], [Paper B] | 高 | ... |
| OP-02 | ... | 推断 ⚠️ | [Paper C] | 低 | ... |
```

### 9.3 绘制方法论图谱

```
操作：

输出格式：
```

### 9.4 生成四类创新方向

基于开放问题池和方法论图谱，从以下四个角度生成创新方向建议：

---

#### 方向类型 1：深化与优化 (Deepen & Optimize)

```
定义：在当前已有方法范式内部进行改进

触发条件：
- 方法论图谱显示某个范式组内有多个相关但未相互比较的方法
- 开放问题池中有关于方法效率、精度、鲁棒性的问题
- 某篇论文的实验暴露了方法的明显弱点

典型表现形式：
- 改进 SOTA 方法的某个模块
- 为现有方法引入正则化/加速/内存优化
- 在更严格或更贴近现实的设定下重新评估现有方法
- 跨数据集/跨领域的鲁棒性验证

输出字段：
- 方向名称
- 灵感来源: [具体文献 + 具体观察]
- 新颖度: 中低（延续而非颠覆）
- 可行性: 高（有成熟基线可比较）
- 风险: 增量式改进可能难中顶会
```

---

#### 方向类型 2：技术迁移 (Method Transfer)

```
定义：将一个领域/场景中的成熟方法迁移到当前场景中

触发条件：
- 另一领域存在与当前场景问题结构相似的方法
- 方法论图谱中显示当前场景缺少某类范式的方法
- 开放问题池显示某些问题在其他领域已有解决方案

典型表现形式：
- 将 NLP 的 prompt learning 迁移到分子生成
- 将 CV 的自监督预训练方法迁移到医疗影像
- 将强化学习的探索策略迁移到材料设计

输出字段：
- 方向名称
- 灵感来源: [源领域方法] + [当前场景的适配性分析]
- 新颖度: 中高（在目标场景中是新的）
- 可行性: 中（需要适配和非平凡改造）
- 风险: 迁移后效果可能不达预期；可能需要证明"为什么该迁移是非平凡的"
```

---

#### 方向类型 3：无人区探索 (Gap Exploration)

```
定义：方法论图谱和开放问题池都显示存在未被探索的空间

触发条件：
- 方法-问题映射矩阵中存在明显的空白单元格
- 多篇论文的开放问题指出了同一方向但无人解决
- 方法论图谱显示存在理论/方法上的断层

典型表现形式：
- 当前所有方法都假设 X，但没有方法考虑 Y 场景
- 某类数据/模态在当前问题中被普遍忽略
- 理论上存在更优解但工程上尚未实现

输出字段：
- 方向名称
- 灵感来源: [方法-问题矩阵中的空白 + 开放问题的交集]
- 新颖度: 高（填补空白）
- 可行性: 低-中（可能需要新方法设计、数据收集等）
- 风险: 高"为什么之前没人做？"需要有说服力的解释
```

---

#### 方向类型 4：问题嫁接 (Problem Grafting)

```
定义：将当前场景的开放问题与另一个不直接相关领域的核心关切结合

触发条件：
- 跨领域的共同底层问题（如公平性、可解释性、效率等）
- 当前场景中未被关注的评估维度
- 社会/产业需求引入的新约束

典型表现形式：
- 在分子性质预测中加入公平性考量（不同分子子类的预测偏差）
- 将因果推断引入推荐系统评估
- 在模型压缩中加入能效约束（绿色 AI 视角）

输出字段：
- 方向名称
- 灵感来源: [嫁接的两个领域/视角 + 结合的逻辑]
- 新颖度: 高（通常是论文中较少见的组合）
- 可行性: 取决于具体方向
- 风险: 可能过于 niche，需要找准发表渠道
```

---

### 9.5 方向筛选与排序

```
对生成的四类方向进行第二轮筛选：

排序依据（加权）：
1. 文献支撑强度 (30%): 该方向是否有开放问题/方法缺口的具体支撑
2. 新颖度 (25%): 在当前文献池中是否有先例
3. 可行性 (25%): 实现门槛（数据、算力、理论储备）
4. 影响潜力 (20%): 如果成功，对领域的推动力

排序后，每类方向至少保留 1 个，总共建议输出 4-8 个方向。
```

### 9.6 分支情况与异常处理

| 情况 | 处理方式 |
|------|---------|
| 某类创新方向无法生成合理建议 | 标注"该方向在当前场景下未产生有证据支撑的建议"，不强行凑数 |
| 多个方向高度相似 | 合并并说明内在联系 |
| 开放问题池极度偏重某一主题 | 如实反映，同时标注"其他维度的开放问题可能未被当前文献池充分覆盖" |

## 11. 设计原则

### 11.1 学科无关性

- 框架本身不内嵌任何学科特有的假设、术语或流程
- 根据用户输入自动识别领域（通过步骤 0 的场景分类和步骤 1 的证据检索），但步骤 1-5 的处理逻辑保持不变
- 不同学科仅在使用不同的文献检索源和术语体系上有差异

### 11.2 防幻觉设计

| 层级 | 策略 |
|------|------|
| 场景级 | 无证据（综述/benchmark/workshop）则明确告知"未找到"，不编造场景 |
| 文献级 | 所有引用的论文必须是真实存在的，筛选标准公开透明 |
| 分析级 | 严格区分"作者自述"与"推断"，前者需标明来源位置，后者需标注 ⚠️ 并提供推理逻辑 |
| 方向级 | 每个创新方向必须链接到至少一篇具体文献的具体观察或开放问题，不凭空生成"看起来合理"的方向 |

### 11.3 分阶段输出

- 本框架的边界清晰：**仅输出创新方向建议，不涉及后续的论文写作、代码实现或实验设计**
- 用户可在获得报告后，自行进入论文写作或实验阶段
- 如果需要下游任务的 skills，应为独立模块，不应与本框架耦合

### 11.4 用户控制权

- 每个步骤之间可以暂停，等待用户确认或调整方向
- 用户可在任何步骤调整检索范围、场景聚焦、文献筛选标准等参数
- 框架提供建议，用户做决策
